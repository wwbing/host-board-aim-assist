# 1. 项目概述                                                                                                                                     
                                                                                                                                                    
  Net_Aim_Control 当前是一个单进程的 Windows 主机端主控程序，最终产物仍然叫 remote_box_receiver.exe。它现在同时承担 5 类职责：                      
                                                                                                                                                    
  - sender 端：启动 ffmpeg.exe，用 ddagrab 抓取桌面中心 ROI，h264_nvenc 编码后通过 UDP + MPEG-TS 推到板子                                           
  - result/control 端：在主机本地 UDP 端口接收板子每帧回传的 JSON 检测结果                                                                          
  - target selection 端：在主机侧做 IDLE / CANDIDATE / TRACKING / LOST 状态机和评分选目标                                                           
  - mouse control 端：把目标点转换成相对屏幕中心偏移 dx/dy，再调用 ddll64.dll 的 MoveR(dx, dy) 做相对移动                                           
  - profiling/timing 端：做 clock offset 校准、结果 timing 解析、anchor 建立、nominal_send_ms 估算和延迟日志/显示                                   
                                                                                                                                                    
  当前主机端已经不是“只收结果”的小工具，而是“发送 + 接收 + 目标决策 + 鼠标控制 + profiling”的统一主控程序。                                         
                                                                                                                                                    
  # 2. 当前主流程                                                                                                                                   
                                                                                                                                                    
  程序启动后的真实顺序是：                                                                                                                          
                                                                                                                                                    
  - main.cpp 解析命令行，打开 DPI awareness，自动探测主屏尺寸，并补齐默认 ROI / sender 参数
  - App::Run() 先初始化 UDP 接收，再初始化鼠标 DLL                                                                                                  
  - 如果开了 --savejson，打开 JSONL 保存文件                                                                                                        
  - 如果没 --disable_sender，启动 sender service                                                                                                    
  - 初始化 profiling：把 sender timing context 交给 profiler，并初始化 clock offset client                                                          
  - 进入主循环：                                                                                                                                    
      - 控制台模式：Sleep(10) 轮询                                                                                                                  
      - --display 模式：走 Win32 消息循环，WM_TIMER 周期 33ms                                                                                       
                                                                                                                                                    
  每轮循环里的核心顺序是：                                                                                                                          
                                                                                                                                                    
  主机 sender(ffmpeg) -> 板子视频输入                                                                                                               
  板子推理 -> UDP JSON 回主机                                                                                                                       
  主机 UdpReceiver::ReceiveLatest() 取最新包                                                                                                        
  -> ParseFrameJson()                                                                                                                               
  -> TimingProfiler::UpdateFrame()                                                                                                                  
  -> UpdateTracker()                                                                                                                                
  -> 若 tracker == TRACKING 且 Q 开启                                                                                                               
  -> MouseController::MoveRelative(dx, dy)                                                                                                          
                                                                                                                                                    
  并行/旁路：                                                                                                                                       
  PollSenderService()                                                                                                                               
  PollProfiling()                                                                                                                                   
    -> clock offset StartMeasurement/Poll                                                                                                           
    -> 每秒打印 profiling 摘要                                                                                                                      
                                                                                                                                                    
  几个容易忽略但很重要的点：                                                                                                                        
                                                                                                                                                    
  - 结果接收是 latest-frame 策略，不排队；每次 ReceiveLatest() 会把 socket 里积压的包全部读掉，只保留最后一个                                       
  - MoveR 只在 tracker_state_ == TRACKING 且 auto_move_enabled_ == true 时触发                                                                      
  - --display 模式和控制台模式不是同一个节拍：前者受 UI 线程和 GDI 绘制影响，后者更适合看 profiling

  # 4. sender 实现总结                                                                                                                              
                                                                                                                                                    
  当前 sender 仍然是外部 ffmpeg.exe 子进程，不是内嵌编码库，也不是自写抓屏线程。实现方式没有改路线，只是做了模块化拆分。                            
                                                                                                                                                    
  当前关键实现点：                                                                                                                                  
                                                                                                                                                    
  - 启动方式：CreateProcessW，并带 CREATE_NEW_PROCESS_GROUP                                                                                         
  - 抓屏：-filter_complex ddagrab=output_idx=...:framerate=...:video_size=...:offset_x=...:offset_y=...                                             
  - 编码：-c:v h264_nvenc                                                                                                                           
  - 低延迟参数：-rc cbr -tune ll -multipass disabled -g ... -forced-idr 1 -bf 0 -rc-lookahead 0 -delay 0 -zerolatency 1
  - 输出：-f mpegts udp://<ip>:<port>?pkt_size=<...>&buffer_size=<...>                                                                              
  - 停止：先 GenerateConsoleCtrlEvent(CTRL_BREAK_EVENT, pid)，超时再 TerminateProcess                                                               
                                                                                                                                                    
  sender 已经完全并到主控项目里，但仍然保持解耦：                                                                                                   
                                                                                                                                                    
  - App 只负责 Start / Poll / Stop                                                                                                                  
  - sender 自己不读键盘，不处理热键                                                                                                                 
  - 第一版没有自动重启                                                                                                                              

  真实配置情况：                                                                                                                                    
                                                                                                                                                    
  - CLI 已暴露：ffmpeg_path、目标 IP/port、framerate、crop、offset、bitrate、maxrate、bufsize、gop、pkt_size、udp_buffer_size                       
  - pts_per_second 仍然在配置和 sender context 里保留，但当前 timing estimator 已经不再依赖它                                                       
  - sender stdout/stderr 虽然建了管道线程，但当前代码把日志 chunk 丢弃了，默认不会把 ffmpeg 内部 warning 打出来                                     
                                                                                                                                                    
  # 5. 检测结果接收与解析                                                                                                                           
                                                                                                                                                    
  主机端接收逻辑：                                                                                                                                  
                                                                                                                                                    
  - UDP 监听地址默认 0.0.0.0:6000                                                                                                                   
  - socket 是非阻塞的                                                                                                                               
  - 每次 ReceiveLatest() 循环 recvfrom() 到 WSAEWOULDBLOCK 为止，只返回最后一个 datagram                                                            
  - 所以当前策略是“只处理最新结果，不保留历史积压”                                                                                                  
                                                                                                                                                    
  当前 JSON 协议里，真正必须的字段是：                                                                                                              
                                                                                                                                                    
  - seq                                                                                                                                             
  - frame_width                                                                                                                                     
  - frame_height                                                                                                                                    
  - box_count                                                                                                                                       
  - boxes                                                                                                                                           
                                                                                                                                                    
  当前可选字段是：                                                                                                                                  
                                                                                                                                                    
  - frame_pts                                                                                                                                       
  - frame_best_effort_ts                                                                                                                            
  - frame_time_base_num                                                                                                                             
  - frame_time_base_den                                                                                                                             
  - timing.board_wall_infer_done_ms                                                                                                                 
  - timing.board_preprocess_ms                                                                                                                      
  - timing.board_inference_ms                                                                                                                       
  - timing.board_postprocess_ms                                                                                                                     
  - timing.board_result_send_start_ms                                                                                                               
  - timing.board_result_send_end_ms                                                                                                                 
                                                                                                                                                    
  向后兼容字段也支持：                                                                                                                              
                                                                                                                                                    
  - timing.result_send_start_ms                                                                                                                     
  - timing.result_send_end_ms                                                                                                                       
  - timing.board_infer_us                                                                                                                           
  - timing.board_post_us                                                                                                                            
                                                                                                                                                    
  boxes 的当前解析规则：                                                                                                                            
                                                                                                                                                    
  - x1/y1/x2/y2 是硬要求，缺任一个就跳过该框                                                                                                        
  - cx/cy 可缺；缺了就用 (x1+x2)/2、(y1+y2)/2                                                                                                       
  - class_id 可缺，默认 -1                                                                                                                          
  - score 可缺，默认 0.0                                                                                                                            
                                                                                                                                                    
  当前解析层的容错风格比较“宽松”：                                                                                                                  
                                                                                                                                                    
  - timing 缺失不会报错                                                                                                                             
  - 非 object 的 box 会被跳过                                                                                                                       
  - box_count != boxes.size() 不报错                                                                                                                
  - malformed box 只会被静默忽略                                                                                                                    
  - --savejson 只保存“已经成功 parse 的 JSON 包”                                                                                                    
                                                                                                                                                    
  补一句重要事实：frame_best_effort_ts 当前只是解析了，没有参与任何后续计算。                                                                       
                                                                                                                                                    
  # 6. 状态机与目标选择                                                                                                                             
                                                                                                                                                    
  当前状态机真实实现就在 App::UpdateTracker() 里，没有拆成独立 tracker 模块。状态有 4 个：                                                          
                                                                                                                                                    
  - IDLE                                                                                                                                            
  - CANDIDATE                                                                                                                                       
  - TRACKING                                                                                                                                        
  - LOST                                                                                                                                            
                                                                                                                                                    
  当前切换逻辑：                                                                                                                                    
                                                                                                                                                    
  - IDLE                                                                                                                                            
    找所有满足 score >= min_score 的框里评分最高的，进入 CANDIDATE，不会单帧直接进 TRACKING                                                         
  - CANDIDATE                                                                                                                                       
    只优先看“与当前候选在 continue gate 内”的延续目标；连续命中到 acquire_confirm_frames 才进入 TRACKING
    如果延续不到，就重新选新的候选；再没有候选就回 IDLE                                                                                             
  - TRACKING                                                                                                                                        
    优先找当前目标延续；如果延续不到，转 LOST                                                                                                       
    同时每帧也会看全局最优框；只有当它比分数当前目标高出 switch_margin，并连续命中 switch_confirm_frames，才允许切换                                
  - LOST                                                                                                                                            
    在 lost_timeout_ms 内尝试按 continue gate 重新找回原目标                                                                                        
    找回则回 TRACKING                                                                                                                               
    超时则回 IDLE                                                                                                                                   
                                                                                                                                                    
  评分函数当前就是 3 项加权和：                                                                                                                     
                                                                                                                                                    
  - center_score                                                                                                                                    
  - confidence_score                                                                                                                                
  - continuity_score                                                                                                                                
                                                                                                                                                    
  真实控制输出是 tracked_target_->selection，不是“当前最新被高亮的框”。ApplyAutoMove() 的 gating 也很直接：                                         
                                                                                                                                                    
  - auto_move_enabled_ == true                                                                                                                      
  - 有 recent packet                                                                                                                                
  - tracker_state_ == TRACKING                                                                                                                      
  - tracked_target_ 存在                                                                                                                            
  - DLL 支持 MoveR                                                                                                                                  
                                                                                                                                                    
  当前坐标转换必须特别注意：                                                                                                                        
                                                                                                                                                    
  - 代码没有做比例缩放                                                                                                                              
  - 它直接用 screen_x = roi_offset_x + round(box.cx)，screen_y = roi_offset_y + round(box.cy)                                                       
  - 然后 dx = screen_x - screen_width/2，dy = screen_y - screen_height/2                                                                            
                                                                                                                                                    
  这意味着当前代码默认假设：板子回传的 frame_width/frame_height 坐标系和主机 sender 抓取 ROI 像素坐标是 1:1 对齐的。只要你一直是 640x640 ROI，这个假  设成立；如果以后 sender crop 和结果 frame 尺寸不一致，就会出偏差。                                                                                
                                                                                                                                                    
  当前鼠标控制语义也很明确：                                                                                                                        

  - 用的是相对控制 MoveR(dx, dy)                                                                                                                    
  - dx/dy 是相对屏幕中心，不是相对当前光标                                                                                                          
  - 这适合游戏内相对视角，不是桌面绝对点鼠标                                                                                                        
                                                                                                                                                    
  # 7. profiling / timing / offset 当前真实状态                                                                                                     
                                                                                                                                                    
  ## 7.1 offset 校准                                                                                                                                
                                                                                                                                                    
  当前主机端 offset client 已经是完整可用实现，且从你最近的运行日志看，这条链路已经打通。                                                           
                                                                                                                                                    
  实现方式：                                                                                                                                        
                                                                                                                                                    
  - 主机端创建一个非阻塞 UDP socket，并 bind(0) 到系统分配的临时本地端口                                                                            
  - 按固定间隔向板子 clock_offset_ip:clock_offset_port 发送 JSON 请求                                                                               
  - 同一个 socket 上 recvfrom() 收响应                                                                                                              
  - 用 NTP 风格四时间戳算 offset 和 RTT/delay                                                                                                       
                                                                                                                                                    
  当前日志里的 3 个关键字段语义是：                                                                                                                 
                                                                                                                                                    
  - offset_state
    当前 offset client 的瞬时状态，比如 request_sent、valid、timeout                                                                                
  - offset_last_valid_ms                                                                                                                            
    最近一次成功样本的 offset 值；只要这个字段有值，就说明 profiling 可以继续拿它做校正                                                             
  - offset_rtt_ms                                                                                                                                   
    最近一次成功样本的 RTT/delay                                                                                                                    
                                                                                                                                                    
  当前 profiling 的使用语义是：                                                                                                                     
                                                                                                                                                    
  - 不是看 offset_state == valid 才能算                                                                                                             
  - 而是看 offset_last_valid_ms 是否存在                                                                                                            
  - 所以 offset_state=request_sent 但 offset_usable=yes 是正常的，表示“当前新请求在路上，但仍沿用上一次有效 offset”                                 
                                                                                                                                                    
  当前 offset 是否容易受显示/UI 干扰：会。                                                                                                          
                                                                                                                                                    
  原因不是 offset client 单独有问题，而是当前整个 App 都在主线程轮询：                                                                              
                                                                                                                                                    
  - 控制台模式下是 10ms loop                                                                                                                        
  - --display 模式下受 WM_TIMER(33ms) 和 GDI 绘制影响                                                                                               
                                                                                                                                                    
  所以如果你是为了看 timing，控制台模式通常比 display 模式更干净。
                                                                                                                                                    
  ## 7.2 nominal_send_ms / anchor 机制
                                                                                                                                                    
  当前之所以需要 anchor，是因为代码已经不再把 sender_start_wall_ms 当 sender timeline 0 点。当前实现默认认为：                                      
                                                                                                                                                    
  - ffmpeg 进程启动时刻不等于第一帧真正进入稳定媒体时间线的时刻                                                                                     
  - 所以只能用“PTS + 板子校准后的时间”建立一个近似锚点                                                                                              
                                                                                                                                                    
  当前 anchor 的建立条件是：                                                                                                                        
                                                                                                                                                    
  - 当前还没有 anchor                                                                                                                               
  - 当前帧有有效 frame_pts                                                                                                                          
  - 当前帧有有效 frame_time_base_num/den                                                                                                            
  - 当前帧有有效 board_wall_infer_done_ms                                                                                                           
  - 当前有可用 offset_last_valid_ms                                                                                                                 
  - 当前帧有 board_pre + board_infer + board_post                                                                                                   
                                                                                                                                                    
  建立后固定保存：                                                                                                                                  
                                                                                                                                                    
  - anchor_frame_pts                                                                                                                                
  - anchor_time_base_num                                                                                                                            
  - anchor_time_base_den                                                                                                                            
  - anchor_adjusted_board_infer_done_ms                                                                                                             
  - anchor_pipeline_delay_ms                                                                                                                        
  - anchor_established_host_wall_ms                                                                                                                 
  - anchor_seq                                                                                                                                      
                                                                                                                                                    
  当前 reset 条件只有这些：                                                                                                                         
                                                                                                                                                    
  - sender_restart                                                                                                                                  
  - time_base_changed                                                                                                                               
  - pts_regression                                                                                                                                  
  - stream_gap，当前阈值是 host 收包间隔 > 500ms                                                                                                    
  - pts_discontinuity，当前阈值是 delta_pts 对应时间 > 2000ms                                                                                       
                                                                                                                                                    
  当前代码语义上，anchor_pts 不应该每帧跟着 frame_pts 变化了；它应当在建立后保持不变，直到命中 reset 条件。是否在你的当前运行环境里已经完全稳定，还 
  需要继续看新日志确认，但代码逻辑已经改成“稳定 anchor”而不是“每帧重建”。                                                                           
                                                                                                                                                    
  当前 nominal_send_ms 的真实语义是：                                                                                                               
                                                                                                                                                    
  - anchor_nominal_send_ms = anchor_adjusted_board_infer_done_ms - anchor_pipeline_delay_ms                                                         
  - 当前帧再按 delta_pts * time_base 从这个固定 anchor 推进                                                                                         
                                                                                                                                                    
  它不是 sender 真实发帧 wall time，只是基于 PTS 和板子时间推回来的估算值。                                                                         
                                                                                                                                                    
  ## 7.3 estimated_pipeline_to_infer_done_ms                                                                                                        

  当前这个指标的计算式是：                                                                                                                          
                                                                                                                                                    
  - adjusted_board_infer_done_ms - nominal_send_ms                                                                                                  
                                                                                                                                                    
  前提必须满足：                                                                                                                                    
                                                                                                                                                    
  - 有 anchor                                                                                                                                       
  - 有当前帧 frame_pts                                                                                                                              
  - 有有效 time base
  - 有可用 offset                                                                                                                                   
  - 有 board_wall_infer_done_ms                                                                                                                     

  当前这个值的语义是：                                                                                                                              
                                                                                                                                                    
  - 基于固定 anchor + PTS 推进后，估算“sender timeline 到 board infer done”的耗时                                                                   
                                                                                                                                                    
  它不是严格意义上的真实端到端延迟，不应该被直接解读为：
                                                                                                                                                    
  - 主机真实发包到板子推理完成的墙钟时间差                                                                                                          
  - 或“网络+编码+解码+前处理+推理”的精确总和                                                                                                        
                                                                                                                                                    
  它当前能说明的是：                                                                                                                                
                                                                                                                                                    
  - 在同一条 sender timeline 估算下，pipeline 大概落在什么量级                                                                                      
  - 跨帧是否大体稳定                                                                                                                                
  - anchor/reset 是否工作正常                                                                                                                       
                                                                                                                                                    
  它当前不能说明的是：                                                                                                                              
                                                                                                                                                    
  - 真正的 sender wall-time emission latency                                                                                                        
                                                                                                                                                    
  ## 7.4 result_return_ms                                                                                                                           
                                                                                                                                                    
  当前计算式是：                                                                                                                                    
                                                                                                                                                    
  - host_result_recv_wall_ms - adjusted_board_infer_done_ms                                                                                         
                                                                                                                                                    
  其中：                                                                                                                                            
                                                                                                                                                    
  - adjusted_board_infer_done_ms = board_wall_infer_done_ms - offset_last_valid_ms                                                                  
                                                                                                                                                    
  所以它表达的是：                                                                                                                                  
                                                                                                                                                    
  - 用最后一次有效 offset 把板子 wall time 拉回主机时间轴后，估算“板子推理完成 -> 主机收到结果”的时间差                                             
                                                                                                                                                    
  当前这个指标可以近似反映结果回传延迟，但要带着误差意识看：                                                                                        
                                                                                                                                                    
  - 板子端 board_wall_infer_done_ms 是毫秒级整数                                                                                                    
  - offset 是通过 RTT 10~12ms 的 NTP 样本估出来的                                                                                                   
  - 主机是轮询收包，不是硬实时时间戳                                                                                                                
                                                                                                                                                    
  所以它偶尔出现轻微负数是可以预期的，通常应解读为：                                                                                                
                                                                                                                                                    
  - offset 样本抖动                                                                                                                                 
  - 毫秒级时间戳取整                                                                                                                                
  - 主机轮询节拍误差                                                                                                                                
                                                                                                                                                    
  如果负得很多，那才是异常；如果只是 -1ms ~ -3ms 这种量级，更像校准噪声，不代表真的“时间倒流”。                                                     
                                                                                                                                                    
  ## 7.5 board_result_send_ms                                                                                                                       
                                                                                                                                                    
  当前主机端代码已经支持这个字段，逻辑也比较完整：                                                                                                  
                                                                                                                                                    
  - 先读 timing.board_result_send_start_ms / end_ms                                                                                                 
  - 如果没有，再回退读 timing.result_send_start_ms / end_ms                                                                                         
  - 最后用 end - start 算 board_result_send_ms                                                                                                      
                                                                                                                                                    
  当前行为是：                                                                                                                                      
                                                                                                                                                    
  - end == start 会显示合法的 0ms                                                                                                                   
  - 只有字段缺失、null、或 end < start 才会显示 N/A                                                                                                 
                                                                                                                                                    
  所以从代码角度说，这条字段映射已经打通了。若运行时仍然经常看到 N/A，更应优先怀疑：                                                                
                                                                                                                                                    
  - 当前收到的实际 payload 里 start/end 有缺失或是 null                                                                                             
  - 或者某些帧 end < start                                                                                                                          
  - 而不是主机端根本没解析这个字段
